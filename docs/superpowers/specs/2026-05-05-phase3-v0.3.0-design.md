# Phase 3 v0.3.0 Design — Agent-Facing Release

**Date:** 2026-05-05
**Scope:** Phase 3.3 (MCP synthesis + graph tools) + Phase 3.4 (hybrid retrieval) from `docs/ROADMAP.md`, integrated with the synthesis-tools direction in `docs/features/mindforge-summarization-tools.md`. Plus a shared `~/.config/mindforge/config.yaml` config file.
**Out of scope:** Phase 3.1 + 3.2 (HTTP API + Web UI) — those land in v0.4.0 once we've dogfooded the agent-facing surface and have real usage signal to drive UI design.

---

## Context

v0.2.1 shipped the day before this spec was written. Yesterday's dogfood validated the dual-use markdown flow: MindForge LLM extraction now actually works against modern Ollama models (`qwen3:30b-a3b` on a GB10), produces 289 typed-relationship concepts, and renders cleanly in Obsidian.

What's missing from "actually useful daily" is the agent-facing surface. Today the MCP server exposes flat lookups (`search`, `get_concept`, `list_concepts`, `get_neighbors`, `get_stats`, multi-KB management). What an agent actually wants is **synthesis** — "what do I know about X?" should return prose paragraphs, not the raw concept files. Returning raw artifacts makes orchestrating-agent context burn quickly on long sessions. Pay the context cost server-side instead.

v0.3.0 ships two coordinated feature blocks:

1. **Hybrid retrieval** — keyword + semantic + graph-walk fusion, default for all queries. Silently lifts CLI search, MCP search, and the synthesis tools' retrieval backend.
2. **MCP synthesis + graph tools** — five new tools, four of which run server-side LLM synthesis to compress responses. `summarize_query` is the headline agent tool.

A small but real third block is needed to keep the surface coherent:

3. **Common config file** — both `mindforge ingest` and `mindforge mcp` now have a long enough flag tail (`--llm-base-url`, `--llm-model`, `--llm-summarize-model`, `--embedding-provider`, retrieval weights, seed pool size) that requiring CLI flags every invocation is hostile. CLI flags continue to work and override the file.

## Decisions made during the brainstorm

These are recorded so the implementation plan doesn't relitigate them:

- **Audience for v0.3.0: the dogfood user (you).** Optimizations bias toward "Claude Code session daily friction" rather than new-user onboarding.
- **Sequencing:** v0.3.0 = agents-facing (this spec). v0.4.0 = humans-facing (HTTP API + Web UI). Decoupled because UI design improves dramatically once we know which queries actually run from agents in practice.
- **Synthesis tools dominate the MCP surface.** `get_context_pack` (an earlier draft from `mcp-extensions.md`) is dropped — `summarize_query` subsumes it with much better context economics (~200-400 tokens vs ~2000 tokens of markdown blob).
- **Single default model, optional override for the headline tool.** On owned hardware the latency / VRAM cost of routing per-tool to two models doesn't pay back. On metered APIs (OpenRouter, Ollama Cloud, OpenAI) the per-token cost difference does — `--llm-summarize-model` provides the override without making the default config complex.
- **Embeddings provider strategy.** Stay an optional extra. Ship two new providers in v0.3.0: an Ollama-native provider (uses `/api/embeddings`) and an OpenAI-compatible provider (covers llama.cpp's `./server --embedding`, OpenAI proper, vLLM, LM Studio). Sentence-transformers stays as the in-process fallback.
- **No write tools.** MCP server is read-only in v0.3.0. Agents that hallucinate would corrupt the KB; revisit if a credible use case emerges.
- **Eval-driven retrieval weight tuning before tag.** Don't ship the docced 0.4/0.4/0.2 weights blindly. Sweep candidate weights against the fixture corpus, ship the winner if it beats the baseline meaningfully; otherwise document the result and ship the baseline.

## Phase 3.4 — Hybrid retrieval

### Design

Per `docs/features/hybrid-retrieval.md`. Three scorers fused per query:

```
keyword (BM25-lite over name + definition + insights)  → score_k  ∈ [0,1]
semantic (cosine over embeddings, if available)        → score_s  ∈ [0,1]
graph-walk (1-2 hop reinforcement from seed pool)      → score_g  ∈ [0,1]

combined = 0.4·k + 0.4·s + 0.2·g                       (default)
        = 0.6·k + 0     + 0.4·g                        (no embeddings fallback)
```

**Graph walk** runs over the union of top-N keyword + semantic seeds (default `N=10` each, ~15-20 unique after dedup). For each seed `s` with seed score `σ(s)`:

- Contribute `σ(s) · 1.0` to `score_g[s]`
- Hop-1 neighbors: `score_g[n] += σ(s) · edge_confidence · 0.5`
- Hop-2 neighbors: `score_g[n] += σ(s) · edge_confidence · 0.2`

Concepts adjacent to multiple strong hits surface even if they're weak keyword matches themselves. This is the differentiator versus plain keyword/semantic retrieval.

### Result shape

```python
@dataclass
class SearchResult:
    concept: Concept
    score_total: float
    score_breakdown: dict[str, float]   # {"keyword": 0.3, "semantic": 0.5, "graph": 0.1}
    matched_via: list[str]              # ["keyword", "graph"], for explainability
```

`score_breakdown` lets downstream consumers (synthesis tools, future UI, MCP) show *why* each result matched.

### CLI surface

```
mindforge query "x"                              # default: --mode hybrid
mindforge query "x" --mode keyword|semantic
mindforge query "x" --weights 0.4,0.4,0.2
```

No breaking change — existing `--mode` values (`keyword`, `semantic`) keep working.

### Embeddings providers

Two new providers ship plus the existing in-process one. Provider extras gating is split by actual dependency footprint:

- **`OllamaEmbeddingProvider`** (stdlib only — `urllib`) — uses Ollama's native `/api/embeddings`. Configurable model (default `nomic-embed-text`). Fastest path for the GB10 already-have-Ollama case. Ships in **core**, no extra needed.
- **`OpenAICompatibleEmbeddingProvider`** (stdlib only — `urllib`) — uses `/v1/embeddings`. Works with llama.cpp's `./server --embedding`, OpenAI proper, vLLM, LM Studio, anything OpenAI-shaped. Ships in **core**, no extra needed.
- **`SentenceTransformerProvider`** continues to require the `[embeddings]` extra for `sentence-transformers` + `faiss` (heavy deps justify the extra gate).

Net effect: a fresh `pip install mindforge-kb` user can opt into semantic retrieval today by pointing at any local Ollama or llama.cpp embedding endpoint. Only the no-server in-process path requires the heavy extra.

CLI flags / config keys: `--embedding-provider {sentence-transformers,ollama,openai-compat}`, `--embedding-base-url`, `--embedding-model`, `--embedding-api-key`.

### Eval hook (gate before tag)

Extend `mindforge eval` with a retrieval-quality mode:

```
mindforge eval --tune-retrieval
```

- Builds relevance judgments from the existing fixture corpus by taking each concept's `name + tags` as the query and treating that concept (plus tag-overlapping concepts) as the gold set. Synthesized judgments are a v1 proxy for a real relevance corpus, which lands in v0.4.0.
- Sweeps candidate weight combinations across keyword/semantic/graph (e.g., 0.0/0.0/0.0 to 1.0/1.0/1.0 in 0.1 steps, normalized to sum to 1.0).
- Reports recall@5 for each weight combo. Picks the winner.
- Optional flag `--apply` writes the winning weights into `~/.config/mindforge/config.yaml` under `[retrieval]`.

If the swept weights beat the docced baseline (0.4/0.4/0.2) by a meaningful margin (≥5% recall@5 lift), ship them. Otherwise stick with the baseline and record the eval result in the v0.3.0 release notes.

## Phase 3.3 — MCP synthesis + graph tools

### Reframe: synthesis vs raw

Tools split into two tiers by who runs the reasoning:

- **Synthesis tier**: server runs an LLM, returns prose. Compresses responses to ~200-400 tokens regardless of KB size. Calling agent gets paragraphs, not concept files.
- **Raw graph tier**: server returns structure. For agents doing visualization, audit, export.

### Synthesis tier (4 tools)

#### `summarize_query` — P0, headline

Accepts a natural-language question, runs hybrid retrieval, traverses up to `max_hops`, reasons over the results via a server-side LLM call, returns compressed prose.

```json
{
  "tool": "summarize_query",
  "input": {
    "question": "string",
    "max_hops": "integer (default: 2)",
    "max_concepts": "integer (default: 5)",
    "focus_tags": ["optional"],
    "include_provenance": "boolean (default: false)"
  }
}
```

Returns:

```json
{
  "answer": "prose, 2-5 sentences",
  "concepts_consulted": ["slug1", "slug2"],
  "confidence": 0.0_to_1.0,
  "suggested_followup": ["optional slug list"],
  "provenance": [{"slug": "x", "transcript": "...", "turns": [12, 13]}]
}
```

`include_provenance: true` adds the `provenance` array; off by default to keep responses small.

Context cost to caller: ~200-400 tokens regardless of KB size.

#### `explain_concept` — P1

Compressed explanation of a single concept. **`brief` mode skips the LLM** — returns the existing distilled `definition + insights[0]` trimmed to budget. `standard` and `detailed` make a fresh LLM synthesis call for paraphrase/expansion. This keeps brief mode sub-second.

```json
{
  "tool": "explain_concept",
  "input": {
    "concept": "string (slug or name)",
    "depth": "brief | standard | detailed (default: standard)"
  }
}
```

Returns:

```json
{
  "slug": "string",
  "explanation": "string",
  "related_slugs": ["top 3-5 by edge weight"]
}
```

Context cost: ~100 tok (brief) / ~200 tok (standard) / ~400 tok (detailed).

#### `compare_concepts` — P2

```json
{
  "tool": "compare_concepts",
  "input": {
    "concepts": ["slug1", "slug2"],
    "aspect": "optional focus dimension"
  }
}
```

Returns `comparison` (prose), `relationship_types` (edge types found between the concepts), `concepts_consulted`. Context cost ~200-300 tokens.

#### `path_between` — P3

```json
{
  "tool": "path_between",
  "input": {
    "from_concept": "string",
    "to_concept": "string",
    "max_hops": "integer (default: 4)"
  }
}
```

Returns `path` (slug chain), `narrative` (LLM prose describing the conceptual chain), `edge_types`, `found`. Unweighted shortest-path for v1. Context cost ~150-250 tokens.

### Raw graph tier (1 tool)

#### `get_subgraph`

```json
{
  "tool": "get_subgraph",
  "input": {
    "center": "string",
    "depth": "integer (default: 1, max: 3)",
    "edge_types": ["optional filter"]
  }
}
```

Returns nodes + edges as JSON plus a markdown rendering. The "give me the graph as data" tool. The previously-proposed `find_path` and `explain_relationship` are dropped — `path_between` and `compare_concepts` cover those use cases with better defaults for agents.

### LLM backend dependency

The MCP server now requires an LLM endpoint for synthesis. New surface:

```
mindforge mcp \
    --llm-base-url http://inpherence01.local:11434 \
    --llm-provider ollama \
    --llm-model qwen3:30b-a3b \
    --llm-summarize-model nemotron-3-super:latest   # optional override
```

**Default model:** `qwen3:30b-a3b`. Already dogfooded in v0.2.1 for ingest; MoE so faster than dense 30B; thinking-fallback already wired up. Zero new validation surface.

**Single-vs-dual-model rationale.** On owned hardware the textbook "small for cheap, big for hard" pattern doesn't pay back: VRAM pressure plus ~5× latency on 123B models makes `summarize_query` painful, and synthesis-from-retrieved-concepts is solidly within 31B's capability anyway. On metered APIs the per-token cost difference matters, so `--llm-summarize-model` lets cloud users route only the headline tool to a bigger model. Most users won't set the override.

**Graceful degradation on LLM unreachable.** At MCP server startup, health-check the LLM endpoint. If unreachable: synthesis tools register as **disabled with explicit error** ("synthesis backend unavailable; use raw `get_concept` or `get_subgraph` instead"); raw tools (and the existing flat-lookup tools) keep working. Inheriting v0.2.1's dogfood lesson: silent fallback is worse than explicit degradation.

## Common config file

### Location

`~/.config/mindforge/config.yaml` (XDG-compliant on macOS / Linux). On Windows, `%APPDATA%/mindforge/config.yaml`. Optional override via `MINDFORGE_CONFIG=<path>` env var.

### Format

YAML. More readable for the nested structure than TOML's table syntax once you have multiple sections with sub-tables.

### Schema

```yaml
# ~/.config/mindforge/config.yaml

llm:
  provider: ollama          # ollama | openai
  base_url: http://inpherence01.local:11434
  model: qwen3:30b-a3b
  keep_alive: -1
  timeout: 120
  # Optional: route the headline summarize_query tool to a bigger model.
  # Comment out for owned-hardware users; useful for metered APIs.
  # summarize_model: nemotron-3-super:latest

embeddings:
  provider: ollama          # sentence-transformers | ollama | openai-compat
  base_url: http://inpherence01.local:11434
  model: nomic-embed-text

retrieval:
  weights:
    keyword: 0.4
    semantic: 0.4
    graph: 0.2
  seed_pool_size: 10        # top-N from each scorer feeding the graph walker
  walk_depth: 2             # 1-2 hops; 3 noisy
```

### Resolution order

1. CLI flags (highest priority — explicit override)
2. Environment variables (e.g., `MINDFORGE_LLM_BASE_URL`)
3. Config file (`MINDFORGE_CONFIG` if set, else default location)
4. Hard-coded defaults (lowest priority)

A `mindforge config show` command prints the merged effective config (good for debugging).

A `mindforge config init` command writes a fully-commented template to the default location, so new users have a starting point to edit.

## SKILL.md and integration docs

### Tier policy (from `docs/features/mindforge-summarization-tools.md`)

`skills/SKILL.md` and each of `docs/integrations/{claude-code,claude-desktop,codex-cli,generic-mcp,hermes-agent,openai-agents-sdk,openclaw}.md` get a Tool surface section reflecting the v0.3.0 surface, with a four-tier policy:

- **Tier 1 — Metadata** (always safe): `get_stats`, `list_concepts`
- **Tier 2 — Targeted retrieval** (when slug is known): `get_concept`, `explain_concept`
- **Tier 3 — Synthesis** (preferred for open-ended questions): `summarize_query`, `compare_concepts`, `path_between`
- **Tier 4 — Raw multi-result** (avoid in long sessions): `search` (cap `top_k` ≤ 3), `get_neighbors`, `get_subgraph`

Decision rule:

> When the goal is *understanding*, use Tier 3 synthesis tools.
> When the goal is *content manipulation*, use Tier 2 direct tools.
> Never call `search` without a `top_k` limit. Default to `top_k: 3`.

### Cowork compatibility note

When Cowork ships, default all queries to `summarize_query` regardless of slug knowledge — non-developer users have no mechanism to manage context budgets. `explain_concept --depth brief` likely the right Cowork default. **Out of scope for v0.3.0 implementation.** Logged here so the future Cowork integration doesn't relitigate.

## PR phasing

**Cluster A — Hybrid retrieval foundation (week 1-2)**

1. **PR-A1:** BM25-lite keyword scorer + graph walker + hybrid orchestrator in `query/engine.py`. Existing keyword/semantic modes become internal scorers. CLI gets `--mode` and `--weights` flags. No new deps. (Largest PR in cluster A.)
2. **PR-A2:** Ollama + OpenAI-compatible embedding providers. New optional providers in the `[embeddings]` extra. Sentence-transformers continues as fallback.
3. **PR-A3:** `mindforge eval --tune-retrieval` weight sweep against fixture corpus. **Gate before tagging v0.3.0** — sweep, decide, document.

**Cluster B — Common config (week 2)**

4. **PR-B1:** `mindforge/config_file.py` + `mindforge config show` + `mindforge config init`. Wire ingest + query CLIs to read merged config. MCP rewrite happens in cluster C.

**Cluster C — MCP synthesis + graph tools (week 2-3)**

5. **PR-C1:** MCP server LLM-config plumbing — new `--llm-*` flags, config-file integration, startup health check, graceful degradation framework. Foundation for all five new tools.
6. **PR-C2:** `summarize_query` (P0). Largest single PR in cluster C; includes server-side composition, response shape, tests.
7. **PR-C3:** `explain_concept` + `compare_concepts` + `path_between` (P1-P3). Share infra from C2.
8. **PR-C4:** `get_subgraph`. Smallest tool PR.

**Release**

10. **PR-R:** `pyproject.toml` `0.2.x → 0.3.0`. Tag, push, watch the three workflows fire (release, binaries, install-smoke). Verify on PyPI.

That's 10 PRs across roughly 2-3 weeks.

## Exit criteria

- `mindforge query "x"` runs hybrid retrieval by default; `score_breakdown` returned in CLI / API output
- `mindforge eval --tune-retrieval` runs end-to-end against fixture corpus, reports recall@5 per weight combo
- `mindforge mcp` exposes the 5 new tools alongside the existing flat-lookup tools
- LLM endpoint reachable → all synthesis tools work; LLM unreachable → synthesis tools register as disabled with explicit error, raw tools work
- `mindforge config show` prints the merged effective config; `mindforge config init` writes a commented template
- `skills/SKILL.md` + 7 integration docs reflect the v0.3.0 tool surface and tier policy
- `pytest` + `ruff check .` + `ruff format --check .` + `mypy mindforge` all green
- v0.3.0 tag → release.yml + binaries.yml + install-smoke.yml all green
- `pip install mindforge-kb==0.3.0` resolves and runs
- Day-after smoke: spend a real Claude Code session using `summarize_query` for retrieval; if it materially reduces context burn, ship-as-default validated

## Open questions deferred to implementation

These are the questions that came up during the brainstorm and were deliberately resolved as "decide during implementation":

- **`compare_concepts` aspect templating.** When `aspect` is set, the LLM prompt should focus on that dimension. Templating it cleanly requires a few example transcripts to dogfood. Decide during PR-C3.
- **`summarize_query` confidence calibration.** Returning `confidence: float` is in the doc but the calibration source is unclear. Options: average of consulted concepts' confidences, retrieval-score-derived, fixed 1.0 if LLM call succeeded. Decide during PR-C2.
- **`mindforge config init` behavior when file exists.** Refuse, prompt, or backup-and-write? Decide during PR-B1.
